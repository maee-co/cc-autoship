# テストルール

コード変更時はテストで品質を担保する。ただし TDD はトークンを 1.5〜2 倍消費するため、**効く領域に絞って適用** する（一律強制で並列ワーカーのスループットを落とさない）。

## TDD 適用領域マトリクス


| 領域 | TDD | unit | e2e | 根拠 |
|------|-----|------|-----|------|
| ロジック（utils / lib / hooks / 純関数） | **必須** | ✅ | - | 既存テスト資産あり・回帰防止効果大 |
| API ルート / バックエンド | **必須** | ✅ | - | input/output 契約を先に固定すると AI の誤解釈が激減 |
| Bash script（`scripts/claude-hooks/lib/*`） | **必須** | ✅ | - | 既に `__tests__/test-*.sh` パターン定着（例: `test-command-match.sh` / `test-team-routing-tag.sh`） |
| AI スキル（`.claude/skills/*`） | **必須** | snapshot | - | 既存パターン踏襲 |
| UI コンポーネント（単体） | 任意 | 推奨 | - | 探索的領域・過剰品質回避 |
| UI 機能（PRD 単位） | - | - | **L1 必須** | 「動くものが本当に動く」を担保（下記「e2e の必須範囲」） |
| 設定変更 / one-shot script | - | - | - | 動作確認のみ |
| インフラ（`.github/workflows/`） | - | - | - | `actionlint` + 手動 |

## TDD 必須領域での進め方

失敗するテストを先に書く厳格 TDD（Red → Green → Refactor）に従う。**必須領域では、実装着手前に Red（失敗するテスト）を先に書き、テストを 1 本も書かずに実装を始めない。**

1. **Red**: 失敗するテストを先に書く（期待される動作を明文化）
2. **Green**: テストを通す最小限の実装
3. **Refactor**: テストが通った状態を保ちつつコード整理

途中で「テストを後から書こう」という誘惑が出たら **必ず red から書き直す**。テストを後付けで通る形に合わせて書くと、AI の「動かしてないのに完了報告する癖」「要件解釈ミス」を遮断できなくなる。


## e2e の必須範囲（UI 機能）

UI 機能（PRD 単位）の e2e は素早く作る × 品質担保のバランスで 3 レイヤに分ける。

| レイヤ | 必須/任意 | 内容 |
|--------|----------|------|
| **L1: golden path** | **必須** | 「PRD の核となる 1 シナリオ」を `e2e/golden-path.spec.ts` 1 ファイルに（1〜3 ケース程度）。UI 変更を含む PR でこれが無ければ auto-merge ブロック |
| **L2: 主要エッジケース** | 推奨 | エラー表示・空状態・認証フォールバック等を `e2e/edge-cases.spec.ts` に。AI が PRD を見て自律判断で追加 |
| **L3: 網羅** | オプション | 後追いで メンテナが指示した場合のみ |

L1 だけで開始し、過剰品質を避けつつ最低限の動作を担保する。


## 共通ルール

- コード変更時はユニットテストを書く or 既存テストを通す（上記マトリクスの「必須」領域は TDD で先にテスト）
- 重要フローは統合テストを検討する
- 外部依存は適切にモックする（fetch 注入等の DI スタイルでテスタブルにする）
- 既存テストを壊さない（変更後に必ず関連テストを green に保つ）
- **外部 API クライアントのテストでは送信ペイロードも assert する**: レスポンスだけでなく、リクエスト側（クエリ文字列・request body・variables）も検証すること。GraphQL の場合は query 文字列と variables の両方を assert する（クエリ変更の無検知を防ぐため — {ISSUE-ID}）
- **hook 系テストは runner 経由でのみ実行する**: `scripts/claude-hooks/__tests__/test-*.sh` は `assert_*` ヘルパーと `$HOOKS_DIR` を runner が注入する設計のため、`bash scripts/claude-hooks/__tests__/test-runner.sh`（引数なしで全件実行）で走らせる。テストファイルの単体実行は `command not found` になる。**TDD 中は `--only <部分一致パターン>` で対象を絞れる**（{ISSUE-ID}。全件 4 分超 → 対象 1 件なら 0.2 秒。マッチ 0 件は exit 1・未知オプションは exit 2 で fail-closed）。Skill 固有テスト（`.claude/skills/*/__tests__/`・`skill-development.md` の単体実行可能ルール）とは設計が異なる。詳細は `scripts/claude-hooks/__tests__/README.md`
