# コードレビュー チェックリスト（全項目）

> 本ファイルは `rules/code-review.md` から移設された詳細仕様。コアルールは元ファイル参照。

## 1. 型安全

- [ ] `any` 型が使われていないこと
- [ ] 適切な型定義・型ガードが使われていること
- [ ] `as` によるキャストが最小限であること
- [ ] ジェネリクスが適切に使われていること

## 2. セキュリティ

- [ ] RLS（Row Level Security）ポリシーが適切に設定されていること
- [ ] ユーザー入力がサニタイズされていること
- [ ] API キー・シークレットがハードコードされていないこと
- [ ] OWASP Top 10 に該当する脆弱性がないこと
- [ ] セキュリティ修正・脆弱性対応の PR は `[codex-review]` を付けて Codex 二次レビューを通していること（付け忘れても hook が意図キーワードで自動検知。`rules/dev-flow.md`「Codex 二次レビュー」参照）

## 3. テストカバレッジ

- [ ] 新規コードにユニットテストがあること
- [ ] エッジケースがテストされていること
- [ ] モックが適切に使われていること
- [ ] 既存テストが通ること
  - *TDD 適用領域・e2e の必須範囲は `rules/testing.md` のマトリクスが SSoT（ロジック/API/Bash/AI スキルは TDD 必須、UI 機能は L1 e2e 必須、UI 単体は任意）*

## 4. パフォーマンス

- [ ] 不要な再レンダリングがないこと（React.memo, useMemo, useCallback）
- [ ] N+1 クエリがないこと
- [ ] 画像が `next/image` で最適化されていること
- [ ] バンドルサイズへの影響が考慮されていること

## 5. コード品質

- [ ] Server Components が優先されていること（Next.js）
- [ ] `"use client"` が必要最小限であること
- [ ] コードが DRY 原則に従っていること（過剰な抽象化は避ける）
- [ ] エラーハンドリングが適切であること
- [ ] ESLint エラーがないこと
- [ ] 外部 API 呼び出しを含む Skill 変更の場合、`SKILL.md` または `references/` に実行例・response 抜粋・`jq` 等の確認方法が残されていること

## 6. 依存・スコープ整合性

- [ ] 変更が担当スコープ内であること
- [ ] 共有パッケージ（`@core/typescript-config` 等）への影響が考慮されていること
- [ ] `turbo.json` の依存関係に影響がないこと

## レビュー対応 PR コメントの標準フォーマット

`/review` の結果を PR にコメントする際は、以下の形式を使う。フォーマットの SSoT は `/review` スキル（`commands/review.md`）であり、本節はその要点を抜粋したもの。差異が出た場合は `/review` 側を正とする。

指摘は **信頼度（実在確度）と重要度（実害の大きさ）を別軸**とし、**重要度 × 信頼度 × スコープ × 対応** の 4 軸で表す（信頼度ルーブリックは `/review` の「ルーブリック」、対応列の値は同ステップ 7）。

### 通常レビュー（`## レビュー結果`）

見出しは必ず `## レビュー結果` で始める（hook が検知して `/auto-merge` をチェーンするため）。

```markdown
## レビュー結果

**対象**: PR #<番号>（<ブランチ名>）

**サマリ**: <1〜2 文の総評。マージ可否の結論を含める>

### 指摘一覧

| #   | 重要度   | 信頼度 | スコープ   | 指摘       | 対応               |
| --- | -------- | ------ | ---------- | ---------- | ------------------ |
| 1   | Critical | 95     | 今回の差分 | {指摘内容} | ✅ 修正済み        |
| 2   | Major    | 85     | 今回の差分 | {指摘内容} | ⚠️ 一部対応        |
| 3   | Minor    | 70     | 今回の差分 | {指摘内容} | ⏭️ 除外（80 未満） |
| 4   | Major    | 90     | 既存問題   | {指摘内容} | 📌 別 Issue        |

### 検証

- type-check: ✅
- lint: ✅
- test: ✅ (X 件 PASS)

### 判定

<auto-merge 可 / メンテナ 手動マージ待ち（理由）>
```

### `--fix`（`## レビュー指摘修正結果`）

```markdown
## レビュー指摘修正結果

| #   | 重要度   | 信頼度 | スコープ   | 指摘       | 対応                  |
| --- | -------- | ------ | ---------- | ---------- | --------------------- |
| 1   | Critical | 95     | 今回の差分 | {指摘内容} | ✅ 修正済み           |
| 2   | Major    | 85     | 今回の差分 | {指摘内容} | ✅ 修正済み           |
| 3   | Minor    | 60     | 今回の差分 | {指摘内容} | ⏭️ 除外（80 未満）    |
| 4   | Minor    | 80     | 既存問題   | {指摘内容} | 📌 別 Issue           |

### テスト結果

- type-check: ✅
- lint: ✅
- test: ✅ (X 件 PASS)
```

### light（`--light`・{ISSUE-ID}）

light 分類 PR（docs-only / test-only / 実コード ≤ 50 行かつ ≤ 3 ファイル）の短縮レビュー。見出しは通常レビューと同じ `## レビュー結果`、本文 2 行目に `<!-- review:light -->` マーカーを置く。**適用範囲**: 6 カテゴリ確認を差分関連カテゴリのみに絞り（対象外は表に「⏭️ 対象外（light）」）、rules 裏取り読み返しとキャリブレーション fixture 参照を省略、実レビューは sonnet subagent に委譲する。**不変**: 4 軸表・信頼度 80 閾値・判定ステータス・auto-merge 連動は維持。Critical/Major を検出したら通常レビューにエスカレする。詳細は `/review`「`--light` オプション」。

- **重要度（Critical / Major / Minor）と信頼度（0-100）は別軸**として併記する（実害の大小と実在確度を混同しない）
- **信頼度 80 未満の指摘は修正せず**「⏭️ 除外（80 未満）」と明記する（誤検知フィルタ。threshold 80 とルーブリックのアンカー `80`/`90` は `/review` で一貫）
- **スコープ列**で「今回の差分 / 既存問題」を区別し、既存問題は原則「📌 別 Issue」とする（本 PR では修正しない）
- 「対応」列は ✅ 完了 / ⏭️ 除外（80 未満）/ ⏭️ スキップ（理由）/ ⚠️ 一部対応 / 📌 別 Issue のいずれかを使う
- テスト結果は最低限 type-check / lint / test の 3 行を含める
