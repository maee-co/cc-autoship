# /session-retro

セッション中の作業パターンを振り返り、**プロセス・インフラの改善提案**を行うスキル。
スキル化・フック化・ルール化・ワークフロー改善・設定改善の5カテゴリで分析する。

## 差別化

| スキル | 焦点 | 出力先 |
|--------|------|--------|
| 外部記録 skill (任意) | 何をやったか（記録） | 外部メモ（この配布キットには未同梱・別途用意した場合のみ） |
| `reflexion:memorize` | エージェントの知識改善 | CLAUDE.md |
| **`/session-retro`** | プロセス・インフラ改善提案 | ターミナル + GitHub Issue |

## 使い方

```bash
# 基本実行
/session-retro

# コミット数を指定
/session-retro --commits 20

# High 優先度の提案を GitHub Issue に起票
/session-retro --issue
```

## 引数

| 引数 | 説明 | デフォルト |
|------|------|-----------|
| `--commits N` | 分析対象のコミット数 | 10 |
| `--issue` | High 優先度の提案を GitHub Issue に起票 | false |

## 推奨タイミング

セッション終了前に以下の順で実行:

```
1. 外部記録 (任意)        → やったことを外部に記録（記録 skill を別途用意した場合のみ）
2. /session-retro         → プロセス改善提案を出力
3. /reflexion:memorize    → 知見を CLAUDE.md に蓄積（任意）
```

## 手順

### ステップ 1: 引数パース

1. `--commits N` を取得（デフォルト: 10）
2. `--issue` フラグの有無を確認

### ステップ 2: ソース収集

以下を並列で収集する。

**2a. Git 情報**
```bash
# 直近のコミットパターン
git log --oneline -N

# 変更ファイルの分布
git diff --stat HEAD~N..HEAD 2>/dev/null || echo "コミット不足"

# 変更ファイル一覧
git diff --name-only HEAD~N..HEAD 2>/dev/null
```

**2b. 既存インフラ一覧**
```bash
# スキル一覧（名前と説明を抽出）
for f in .claude/skills/*/SKILL.md; do
  name=$(head -5 "$f" | grep "^name:" | sed 's/name: //')
  desc=$(head -5 "$f" | grep "^description:" | sed 's/description: //')
  echo "- $name: $desc"
done

# フック一覧
jq '.hooks // {}' .claude/settings.json

# ルール一覧
ls .claude/rules/*.md
```

### ステップ 3: 会話コンテキスト分析

セッション中の会話を振り返り、以下の観点で分析する。
**これがこのスキルの核心であり、LLM の自己分析能力に依存する。**

#### 検出すべきパターン

1. **繰り返し操作**: 同じ種類のコマンド・ツール呼び出しを複数回行った
2. **定型作業**: 毎回同じ手順で行った作業フロー
3. **手動検証**: 自動化できたはずの検証・確認を手動で行った
4. **試行錯誤**: エラー → 修正 → 再試行のループが発生した
5. **暗黙のルール**: 明文化されていないが従っていた規約やパターン
6. **未使用スキル**: 既存スキルで対応できたが使わなかった場面
7. **ワークフロー逸脱**: dev-flow.md のフローから外れた作業

### ステップ 4: 5カテゴリで提案を生成

収集データと会話コンテキストを突き合わせ、以下の5カテゴリで提案を生成する。

| カテゴリ | 検出対象 | 判断基準 |
|---------|---------|---------|
| **skill-candidate** | 繰り返し操作のスキル化 | 2回以上繰り返した定型作業 |
| **hook-candidate** | 自動実行すべき検証・変換 | 毎回手動で行った検証、ツール実行後の定型処理 |
| **rule-candidate** | 暗黙のルール / 違反パターン | 明文化されていないが従った規約 |
| **workflow-improvement** | 開発フロー改善・既存スキル活用改善 | フロー逸脱、未使用スキル |
| **settings-tweak** | settings.json の設定改善 | 権限不足/過剰、deny ルールの追加 |

#### 重複チェック

- 提案がステップ2bで収集した既存インフラと重複する場合:
  - 完全重複 → 提案から除外
  - 部分重複（既存スキルの拡張） → `workflow-improvement` に分類し「既存スキルの活用改善」として提案

### ステップ 5: レポート出力

以下のフォーマットでターミナルに出力する。

```markdown
## Session Retro レポート

### サマリー
- セッション日時: {YYYY-MM-DD}
- 分析対象: 会話コンテキスト + git {N} コミット
- 検出提案数: {合計} 件（skill: {n}, hook: {n}, rule: {n}, workflow: {n}, settings: {n}）

### 提案一覧

#### 1. [{カテゴリ}] {提案タイトル}

**観測**: {セッション中に観測した具体的なパターン}
**提案**: {改善案の具体的な内容}
**優先度**: High / Medium / Low
**実装見積もり**: Small / Medium / Large

---
（提案ごとに繰り返し）

### 次のアクション

即実行を推奨する提案:
1. {最も優先度の高い提案とその理由}
```

### ステップ 6: GitHub Issue 起票（`--issue` 時のみ）

`--issue` フラグが指定された場合、**High 優先度の提案のみ** GitHub Issue に起票する。

```bash
gh issue create \
  --title "[infra] session-retro: {提案タイトル}" \
  --body "{提案の詳細（観測・提案・カテゴリ・実装見積もり）}" \
  --label "enhancement"
```

- 複数の High 提案がある場合は、それぞれ個別の Issue として起票する
- Issue 作成後、Issue URL をレポートに追記する

## エッジケース対応

| ケース | 対応 |
|--------|------|
| コミットゼロ（調査セッション） | Git 分析をスキップ、会話コンテキストのみで分析 |
| 提案が 0 件 | 「現在のセットアップは十分最適化されています」と報告 |
| 既存スキルと重複する提案 | workflow-improvement に分類し「活用方法の改善」として提案 |
| 非常に短いセッション | 分析に十分なデータがない旨を報告 |

## 安全ルール

- **提案のみ行い、実装はしない**（Edit/Write を allowed-tools に含めない）
- スキル/フック/ルールの作成・変更はユーザーの判断に委ねる
- `--issue` での Issue 起票は メンテナ許可不要（MEMORY.md のルールに準拠）
- `.env` やトークン情報をレポートに含めない
