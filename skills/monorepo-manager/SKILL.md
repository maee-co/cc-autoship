# /monorepo-manager

リポジトリの整合性と開発フロー遵守をチェックする（モノレポでは依存・スコープ整合も検査。単一リポでは A 系は N/A になり B 系のフロー遵守のみ機能する）。
計画レビュー時・commit-push 前に自動実行される。任意タイミングでも `/monorepo-manager` で呼び出し可能。

## 実行モード

引数でモードを切り替える。引数なしの場合は両方実行。

| モード | 引数 | 用途 |
|--------|------|------|
| 計画レビュー | `plan` | 方針提案フェーズで実装計画をチェック |
| コミット前チェック | `pre-commit` | commit-push 前に差分をチェック |
| フルチェック | (なし) | 両方実行 |

## 軽微変更のスキップ基準

以下のいずれかに該当する場合、**チェック内容**をスキップしてよい（worktree + PR は常に必須。hook で main 直接コミットはブロックされる）:
- 変更行数が **5行以下** かつ typo・文言修正のみ
- `.md` ファイルのみの変更
- コミットメッセージが `docs(*)` または `chore(*)` で始まる軽微な更新

## A. モノレポ / マルチパッケージ整合性チェック（単一リポでは N/A）

### A1. スコープ内編集の確認

1. 現在のブランチ名からスコープを特定する
   ```bash
   git branch --show-current
   ```
   - `feat/{ISSUE-ID}-{scope}-{desc}` または `fix/{ISSUE-ID}-{scope}-{desc}` → scope 部分を抽出
   - ISSUE-ID は GitHub Issue 番号（例: `99`）または外部 tracker ID（Linear なら `ABC-99` 等）
   - 複合スコープ（`feat/99-scope1,scope2-desc`）の場合はカンマ区切りで複数スコープとして扱う
   - `main` ブランチの場合 → スコープチェックは **スキップ**（B1 で警告を出す）

2. 変更ファイル一覧を取得（ブランチに応じて分岐）
   ```bash
   # feature/fix ブランチの場合
   git diff --name-only main...HEAD

   # main ブランチの場合（ステージ済み + 直近コミットの差分）
   git diff --name-only --cached
   # ステージ済みがなければ直近コミット
   git diff --name-only HEAD~1..HEAD
   ```

3. 変更ファイルが以下のスコープに収まっているか確認:
   - `apps/{scope}/` — アプリ固有の変更
   - `packages/` — 共有パッケージ（影響範囲の確認が必要、A2 へ）
   - `.claude/` / `docs/` / ルート設定ファイル — `infra` スコープのみ許可
4. スコープ外の変更がある場合は **警告** を出す

### A2. 共有パッケージ影響の確認

1. `packages/` 配下に変更があるか確認
2. 変更がある場合:
   - どのパッケージか特定（`packages/ui/`, `packages/typescript-config/` 等）
   - そのパッケージの **オーナー** を CLAUDE.md の「共有パッケージ オーナー」から照合
   - 依存しているアプリを `package.json` の `dependencies` / `devDependencies` から特定
3. 影響がある場合は **注意** として報告

### A3. 依存関係の整合性

1. `package.json` の変更がある場合:
   - 新規依存の追加が妥当か（重複パッケージの有無）
   - workspace プロトコル（`workspace:*`）が正しく使われているか
2. `turbo.json` の変更がある場合:
   - タスク定義の構造が壊れていないか確認（Read で内容を読み取り検証）

### A4. ディレクトリ構成の確認

1. 新規ディレクトリ・ファイルが CLAUDE.md のディレクトリ構成に沿っているか
2. アプリは `apps/` 配下、共有コードは `packages/` 配下にあるか

## B. 開発フロー遵守チェック

### B1. ブランチ命名規則

1. 現在のブランチ名を取得
2. 以下のパターンに合致するか確認:
   - feature: `feat/{ISSUE-ID}-{scope}-{説明}`
   - bugfix: `fix/{ISSUE-ID}-{scope}-{説明}`
   - `main` ブランチでの直接作業は **警告**（hook で commit/push がブロックされるため worktree 必須）

### B2. コミットメッセージ形式

1. 未マージコミットのメッセージを取得（ブランチに応じて分岐）
   ```bash
   # feature/fix ブランチの場合
   git log main..HEAD --format="%s"

   # main ブランチの場合（直近のコミットを確認）
   git log -1 --format="%s"
   ```
2. 各メッセージが `type(scope): 説明` 形式か確認
   - type: `feat` / `fix` / `docs` / `refactor` / `test` / `chore`
   - scope: アプリ名 / パッケージ名 / `infra`
   - 複合スコープ可: `type(scope1,scope2): 説明`
3. 形式違反があれば **警告** と正しい例を提示

### B3. Issue 紐付け

1. ブランチ名から Issue ID を抽出（`feat/42-...` → `42`、`feat/ABC-42-...` → `ABC-42`）
2. 対応する GitHub Issue を `gh issue list --search "<ID>"` で検索し、存在を確認
3. Issue が見つからない場合は **注意**（外部 tracker → GitHub 同期の遅延の可能性あり）
4. `main` ブランチの場合はこのチェックを **スキップ**

### B4. worktree 運用の確認（計画レビュー時のみ）

1. 現在の作業ディレクトリが worktree 内かを確認する
   ```bash
   git worktree list
   ```
   - 出力が2行以上（メイン + worktree）で、現在のディレクトリが worktree パスと一致していれば OK
2. `main` ブランチで直接作業しており、かつ worktree 外の場合は **注意** を出す
3. 軽微変更の場合はスキップ

## 計画レビューモード（plan）での追加チェック

会話コンテキスト内の方針提案テキストに対して以下を確認する:

1. **変更予定ファイル** がスコープ内か（A1 を計画ベースで実施）
2. **テスト方針** が含まれているか
3. **Issue 作成予定** があるか（軽微変更スキップ基準に該当する場合を除く）
4. **worktree 作成** が計画に含まれているか

計画に不備がある場合は、修正すべき点を具体的に指摘する。

---

## 注意事項

- このスキルは **ブロッカー** ではなく **アドバイザー** として機能する
- ❌（BLOCK）は メンテナの判断で override 可能
- 軽微変更スキップ基準に該当する場合はチェック全体をスキップしてよい
- Agent Teams 運用時は各 Teammate の commit 前にも実行される
