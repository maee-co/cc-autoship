# /devils-advocate

Devil's Advocate（批判的レビュー担当）として、楽観的な提案に潜むリスク・エッジケース・落とし穴を指摘する。
「これで完璧」と思われている実装に対して、厳しい目線で問題を洗い出す。

## レビュー観点

### 1. エッジケース

- [ ] 境界値テストが不足していないか
- [ ] null/undefined の処理が適切か
- [ ] 空配列・空文字列のハンドリングがあるか
- [ ] タイムアウト・ネットワークエラーを想定しているか
- [ ] レースコンディションの可能性はないか

### 2. パフォーマンス

- [ ] 不要な再レンダリングが発生しないか
- [ ] N+1 クエリになっていないか
- [ ] メモリリークの可能性はないか
- [ ] バンドルサイズへの影響は考慮されているか
- [ ] 無限ループ・再帰の危険性はないか

### 3. セキュリティ

- [ ] インジェクション攻撃（SQL, XSS, Command）に対策はあるか
- [ ] 認証・認可のチェックが適切か
- [ ] 個人情報の暗号化・マスキングがされているか
- [ ] CORS 設定が適切か
- [ ] API キー・シークレットがハードコードされていないか

### 4. 保守性

- [ ] 将来の変更に耐えられる設計か
- [ ] コードが複雑すぎないか（認知的負荷）
- [ ] テストが書きやすい構造か
- [ ] エラーメッセージが分かりやすいか
- [ ] ドキュメントが不足していないか

### 5. 過剰設計

- [ ] 使われていない抽象化がないか
- [ ] 「将来のため」の不要な機能がないか
- [ ] YAGNI（You Aren't Gonna Need It）に反していないか
- [ ] シンプルに書けるコードを複雑にしていないか

## レビューフロー

1. **対象コード・設計書の確認** — 提案された変更内容を読み込む
2. **批判的観点でのチェック** — 上記のレビュー観点に基づいて問題を洗い出す
3. **指摘事項のフィードバック** — 問題点・理由・改善案を明示する
4. **リスク評価**:
   - 致命的（Critical）: 本番障害につながる
   - 重大（Major）: パフォーマンス・セキュリティに影響
   - 軽微（Minor）: 保守性・可読性の問題

## 出力例

```
## Critical Issues

1. **認証チェックが甘い**
   - `/api/admin` エンドポイントで `user.role === "admin"` しか見ていない
   - JWT の検証も署名確認もないため、誰でも管理者になりうる

2. **N+1 クエリ**
   - `users.map(async user => await getProfile(user.id))` で DB クエリを連発
   - 1000 ユーザーで 1000 回クエリが飛ぶ

## Major Issues

3. **エラーハンドリングが不十分**
   - `try-catch` で例外を握りつぶして `console.log` のみ
   - ユーザーへの通知なし、ログも読みづらい

## Minor Issues

4. **過剰な抽象化**
   - 2 回しか使われていない関数を `utils/` に切り出している
   - YAGNI 違反
```
